A-Auth

: A Developer friendly authentication protocol

Description


A-Auth derives it's inspiration from O-Auth 2.0. The primary obective of this feature is to provide a standard for naive coders to use an easy-to-use developer friendly authentication mechanism, which achieves the same goal but with much more abstraction.

It is a simpler way to publish and interact with protected data. It's also a safer and more secure way for people to give you access. We've kept it simple to save you time.

If you're storing protected data on your users' behalf, they shouldn't be spreading their passwords around the web to get access to it. Use A-Auth to give your users access to their data while protecting their account credentials.











Advantages of our system

  • Developer Friendly
  • hassle free exchange of tokens
  • Multi level authorization by specifying scope

Design and Implementation




A new layer of abstraction will certainly help developers to regsiter, create applications and authenticat with less hassle about token exchanges, state codes, time stamps expiry, scope of authorization, etc.

The simplified authentication flow has the following steps.

  1. Create an application by resgitering into the A-Auth portal
  2. You are allocated a [client_id, client_secret_key]
  3. Now when you want users to authenticate, send a GET request by appending paramters [client_id, state, scope]
  4. Now the server response will have an acess_code and a hash ( which acts as a state)
  5. The hash serves as a digital signature for your authentication.
  6. Send one more POST request to the server by appending paratmers [client_id, client_secret_key, access_code]
  7. The server responds with an access_token
  8. This is your final token, use this token for future queries




Roles :
  • The Third-Party Application: "Client"
  • The API: "Resource Server"
  • The Authorization Server
  • The User: "Resource Owner"
